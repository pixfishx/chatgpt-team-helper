# 上下游渠道配置指南

本文用于指导管理员完成以下几类配置：

- 新增一个可访问的渠道入口，例如 `/redeem/douyin`
- 让本站在兑换时调用外部平台完成履约
- 让本站作为“上游供应方”对外提供标准卡密接口
- 开启 `/downstream` 下游公开售码页

> 说明：以下配置入口均要求管理员具备 `super_admin` 权限。

## 1. 先理解这 4 个模块

上下游渠道不是只配一个地方就能跑通，通常要同时处理下面 4 个模块：

| 模块 | 作用 | 入口 |
| --- | --- | --- |
| 渠道管理 | 定义渠道 key、兑换模式、Provider、是否参与下游库存 | `/admin/settings` -> `核心与安全` -> `渠道管理` |
| 上下游接口配置 | 配置出站履约和入站 API | `/admin/settings` -> `上下游接口配置` |
| 兑换码库存 | 准备真实可用的库存 | `/admin/redemption-codes` |
| 下游售码配置 | 打开 `/downstream` 公开售码页并设置价格 | `/admin/settings` -> `支付与财务` -> `下游售码配置` |

如果你还希望“支付商城”也能走某个渠道，额外还要去：

- `/admin/settings` -> `支付与财务` -> `支付商品管理`

把该渠道 key 加入商品的 `codeChannels`。

## 2. 推荐配置顺序

建议按下面顺序操作：

1. 先建渠道
2. 再配上下游接口
3. 再补库存
4. 最后开启 `/downstream`
5. 用真实测试单做联调验证

这样最不容易出现“页面能打开但没库存”或“渠道建好了但兑换失败”的情况。

## 3. 第一步：配置渠道

进入：

- `/admin/settings`
- 选择 `核心与安全`
- 找到 `渠道管理`

点击“新增渠道”后，重点关注以下字段：

| 字段 | 含义 | 建议 |
| --- | --- | --- |
| 渠道 Key | 渠道唯一标识，只允许小写字母、数字、连字符 | 例如 `douyin`、`partner-a` |
| 渠道名称 | 后台展示名称 | 例如 `抖音渠道` |
| 兑换模式 | 决定这个渠道如何兑换 | 对接外部卡密请选 `external-card` |
| 履约 Provider | 决定 external-card 如何调用外部系统 | `custom-http` 或 `platform-upstream` |
| 允许回退通用码 | 渠道库存不足时，是否允许回退使用 `common` 渠道库存 | 只在你明确需要共用通用库存时开启 |
| 允许参与下游售码 | 该渠道是否进入 `/downstream` 共享库存 | 要做下游售码时开启 |
| 启用 | 渠道是否生效 | 建议配置完成前先关闭，联调通过后再打开 |
| 排序 | 前台/后台排序参考值 | 数字越小越靠前 |

### 3.1 兑换模式怎么选

- `code`：站内兑换码，适合本地库存
- `external-card`：外部卡密模式，适合对接上游平台或第三方卡密接口
- `linux-do` / `xhs` / `xianyu`：属于内置业务渠道，一般按对应业务文档单独配置

### 3.2 Provider 怎么选

只有 `external-card` 渠道才需要选择外部 Provider：

- `custom-http`
  - 适合“只有一个自定义接口”的第三方平台
  - 系统会在兑换时直接发送 `POST JSON`
  - 不支持标准化的“卡密检查”
- `platform-upstream`
  - 适合同项目实例之间互相对接
  - 统一走标准路径：
    - `GET /api/upstream/health`
    - `POST /api/upstream/cards/check`
    - `POST /api/upstream/cards/redeem`

> 注意：`external-card` 渠道最终使用哪个 Provider，以“渠道管理”里的 Provider 选择为准；而“上下游接口配置”页面用于填写该 Provider 对应的全局参数。

### 3.3 新建渠道后会得到什么

新增并启用后，系统会自动提供一个通用入口：

- `/redeem/<channelKey>`

例如渠道 key 为 `douyin`，则入口为：

- `/redeem/douyin`

## 4. 第二步：配置上下游接口

进入：

- `/admin/settings`
- 选择 `上下游接口配置`

该页面分为两部分：

- 出站履约：本站调用别人的平台
- 入站接口：别人调用本站

### 4.1 本站域名（公网 Base URL）

页面顶部的“本站域名”建议优先填写，例如：

```text
https://example.com
```

它会同时影响：

- 平台通用接口出站时自动携带的下游域名标识
- 支付/开放账号等回调地址生成

如果这里留空，系统会退回到请求上下文或环境变量 `PUBLIC_BASE_URL` 推导。

### 4.2 配置出站履约

出站履约是 `external-card` 渠道的总开关。

只要这里关闭，所有 `external-card` 渠道都不会再向外部系统发请求。

#### 方案 A：自定义接口（custom-http）

适合对接“单个 JSON 接口”的第三方平台。

需要填写：

- 出站履约开关：启用
- Provider 类型：`custom-http`
- 供应商名称：可选，仅用于后台识别
- 请求超时：建议先用默认 `15000`
- 自定义接口 URL：完整的 `http(s)` 地址
- 自定义 Body JSON：必须是合法 JSON

Body JSON 支持 3 个占位符：

- `{{email}}`
- `{{code}}`
- `{{channel}}`

示例：

```json
{
  "userEmail": "{{email}}",
  "cardCode": "{{code}}",
  "channel": "{{channel}}"
}
```

建议：

- 如果上游平台只提供“直接充值/直接兑换”接口，用 `custom-http`
- 如果你还需要标准化的卡密可用性检查，不要选 `custom-http`，改用 `platform-upstream`

#### 方案 B：平台通用接口（platform-upstream）

适合同类站点之间互联。

需要填写：

- 出站履约开关：启用
- Provider 类型：`platform-upstream`
- 平台 Base URL：上游平台根地址，例如 `https://supplier.example.com`
- 出站 API Key：如果上游启用了入站鉴权，则这里填写对方分配给你的密钥
- 请求超时：建议先用默认 `15000`

系统会自动拼接固定路径：

- `GET /api/upstream/health`
- `POST /api/upstream/cards/check`
- `POST /api/upstream/cards/redeem`

### 4.3 配置入站接口

如果你希望“其他平台调用本站库存”，就需要配置入站接口。

在 `入站接口` 页签中：

1. 打开“入站接口开关”
2. 为每个下游实例新增一条规则
3. 填写“下游域名”和“入站 API Key”

规则说明：

- 域名按主机名精确匹配
- 可以填写 `partner.example.com`
- 也可以填写完整 URL，保存时会自动规范化为主机名
- 域名留空表示“默认规则”，会匹配没有单独配置的下游

建议：

- 每个下游实例单独分配一条独立 API Key
- 不要让多个正式下游共用同一条默认规则

### 4.4 入站接口固定路径

入站接口路径固定，不需要额外自定义：

- `GET /api/upstream/health`
- `POST /api/upstream/cards/check`
- `POST /api/upstream/cards/redeem`

如果你配置了“按域名区分”的入站规则，请求方最好显式带上：

- `X-Upstream-Key`
- `X-Downstream-Domain`

其中：

- `X-Upstream-Key`：你在规则里给对方分配的 API Key
- `X-Downstream-Domain`：请求方自己的域名，例如 `reseller.example.com`

虽然系统也会尝试从 `Origin` / `Referer` 推导来源域名，但服务端对服务端联调时，显式传 `X-Downstream-Domain` 最稳定。

### 4.5 入站接口联调示例

#### 健康检查

```bash
curl -X GET 'https://supplier.example.com/api/upstream/health' \
  -H 'X-Upstream-Key: your-api-key' \
  -H 'X-Downstream-Domain: reseller.example.com'
```

#### 卡密检查

```bash
curl -X POST 'https://supplier.example.com/api/upstream/cards/check' \
  -H 'Content-Type: application/json' \
  -H 'X-Upstream-Key: your-api-key' \
  -H 'X-Downstream-Domain: reseller.example.com' \
  -d '{
    "code": "TEST-CODE-001",
    "channel": "douyin"
  }'
```

#### 卡密兑换

```bash
curl -X POST 'https://supplier.example.com/api/upstream/cards/redeem' \
  -H 'Content-Type: application/json' \
  -H 'X-Upstream-Key: your-api-key' \
  -H 'X-Downstream-Domain: reseller.example.com' \
  -d '{
    "code": "TEST-CODE-001",
    "email": "buyer@example.com",
    "channel": "douyin"
  }'
```

> 补充：`/downstream` 售出的“映射码”也走这套入站接口，所以对接方不需要区分“真实码”还是“下游公开售码生成的映射码”。

## 5. 第三步：准备库存

只有配了渠道还不够，必须准备可用库存。

进入：

- `/admin/redemption-codes`

### 5.1 普通站内渠道（`code`）

普通 `code` 渠道应通过站内兑换码方式补货。

适合：

- 账号已在本地管理
- 兑换后由本站直接完成邀请/开通

### 5.2 外部卡密渠道（`external-card`）

`external-card` 渠道不要用普通“批量生成兑换码”补货。

正确做法是：

- 使用“导入外部卡密”功能导入上游卡密

这是因为 `external-card` 渠道的库存本质上是“外部卡密原文”，系统会在兑换时再调用外部 Provider 完成履约。

## 6. 第四步：开启下游公开售码

如果你要把部分库存公开出售给下游用户，还需要配置 `/downstream`。

进入：

- `/admin/settings`
- 选择 `支付与财务`
- 找到 `下游售码配置`

需要配置：

- 启用下游公开售码页
- 商品名称
- 统一售价
- 至少一种支付方式：支付宝 / 微信

公开页面固定为：

- 下单页：`/downstream`
- 查单页：`/downstream/order`

### 6.1 开启 `/downstream` 的前置条件

除了“下游售码配置”本身，还要同时满足：

1. 支付功能已开启
2. ZPAY 已正确配置
3. 至少有一个渠道开启了“允许参与下游售码”
4. 该渠道下存在可售库存

如果 ZPAY 没有配置，下游下单会直接报：

- `支付未配置，请联系管理员`

### 6.2 哪些库存会进入 `/downstream`

并不是所有兑换码都会进入下游共享库存，至少要满足：

- 渠道处于启用状态
- 渠道开启了“允许参与下游售码”
- 渠道兑换模式是 `code` 或 `external-card`
- 兑换码未使用
- 兑换码未被其他订单占用
- 兑换码未被标记为已通过下游售出

额外还有两类特殊限制：

- `code` 渠道
  - 对应账号必须处于开放状态
  - 当前人数不能超限
  - 当前实现只会把“当天创建”的可售账号纳入下游共享库存
- `external-card` 渠道
  - 上游状态不能是 `invalid`
  - 不能是 `used`
  - 不能是 `processing`

如果你发现 `/downstream` 库存始终是 0，优先从上面几项排查。

## 7. 常见配置组合

### 7.1 只做本站自营渠道

适合：

- 只想增加一个新的兑换入口
- 不需要对接第三方接口

建议：

- 渠道模式选 `code`
- Provider 保持 `local`
- 按站内兑换码方式补库存

### 7.2 本站作为下游，调用别人平台

适合：

- 手里没有本地账号库存
- 兑换时需要把卡密提交给上游系统处理

建议：

- 渠道模式选 `external-card`
- Provider 选 `custom-http` 或 `platform-upstream`
- 在“出站履约”里完成参数配置
- 用“导入外部卡密”补库存

### 7.3 本站作为上游，对外供货

适合：

- 其他站点来调用你的库存
- 想开放标准的卡密检查/兑换接口

建议：

- 在“入站接口”里开启 API
- 给每个下游单独配置域名和 API Key
- 用固定路径联调 `/api/upstream/*`

### 7.4 本站既做上游又做下游

适合：

- 多站点互为供货方
- 一个站负责售卖，另一个站负责库存

建议：

- 渠道使用 `external-card`
- Provider 优先使用 `platform-upstream`
- 双方都填写 `本站域名`
- 双方互相分配入站 API Key

## 8. 联调验收清单

建议至少做一次完整验收：

1. 访问 `/redeem/<channelKey>`，确认页面能正常打开
2. 使用一张测试卡密走完整兑换流程
3. 如果是 `platform-upstream`，先测 `/api/upstream/health`
4. 再测 `/api/upstream/cards/check`
5. 最后测 `/api/upstream/cards/redeem`
6. 如果渠道参与下游售码，确认 `/downstream` 能看到库存和支付方式
7. 支付完成后，到 `/downstream/order` 查询订单并确认映射码已发放

## 9. 常见问题排查

### 9.1 渠道建好了，但 `/redeem/<key>` 还是不能正常用

优先检查：

- 渠道是否已启用
- `channelKey` 是否写错
- 前端是否刷新到了最新运行时配置

### 9.2 external-card 渠道兑换时报“服务暂不可用”

优先检查：

- 出站履约总开关是否打开
- 当前渠道 Provider 是否和你实际配置一致
- `custom-http` 的 URL / Body JSON 是否合法
- `platform-upstream` 的 Base URL 是否正确

### 9.3 入站接口返回 `Unauthorized: Unknown downstream domain`

说明当前请求来源域名没有匹配到任何规则。

排查顺序：

1. 请求头里是否显式传了 `X-Downstream-Domain`
2. 该域名是否和后台配置完全一致
3. 是否只配置了精确域名，但没有默认规则

### 9.4 入站接口返回 `Unauthorized: Invalid upstream API key`

优先检查：

- `X-Upstream-Key` 是否正确
- 是否把别的下游的 API Key 用错了
- 规则保存后是否已刷新到最新配置

### 9.5 `/downstream` 页面能打开，但库存一直是 0

优先检查：

- 渠道是否开启“允许参与下游售码”
- 渠道是否处于启用状态
- 是否真的有未使用库存
- 外部卡密是否被标记成 `invalid` / `used` / `processing`
- 普通 `code` 渠道对应账号是否满足当前可售条件

### 9.6 `/downstream` 下单时报“支付未配置，请联系管理员”

说明下游售码页面已开启，但支付侧没配完。

需要补齐：

- `/admin/settings` -> `支付与财务` -> `ZPAY 支付配置`

### 9.7 渠道已经建好了，但支付商城还是没有使用它

这通常不是渠道本身的问题，而是商品策略没有更新。

排查：

- 进入 `/admin/settings` -> `支付与财务` -> `支付商品管理`
- 检查对应商品的 `codeChannels`
- 确认你新建的渠道 key 已加入该商品的渠道优先级列表

## 10. 最后建议

如果你是第一次配置，最稳妥的做法是先走一个最小可用闭环：

1. 新建一个 `external-card` 测试渠道
2. 先用 `platform-upstream` 对接一套测试站
3. 用 1 张测试卡密做联调
4. 跑通后再批量导入库存
5. 最后再开放 `/downstream`

这样能最快定位问题到底出在渠道、接口、库存还是支付环节。
